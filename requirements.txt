# Claude Code Prompt: 搭建 OpenTelemetry Java 微服务可观测性 Demo（增强版）

请帮我创建一个完整的本地 Docker Compose 可观测性演示项目，用于验证 OpenTelemetry + Grafana 全家桶的 metrics/traces/logs 三大信号采集、存储、展示和关联跳转。这个 demo 需要涵盖真实微服务场景，包含 HTTP 跨服务调用、MySQL 数据库查询、Redis 缓存操作，展示 OTel Java Agent 的零代码自动 instrumentation 能力。此外每个服务需要预留故障注入 API，可以在运行时动态注入延迟、异常和错误率，用于模拟各种故障场景做可观测性演练。

### 2. 故障注入引擎（ChaosMonkey）— 两个服务共用同一套设计

这是 demo 的核心特性之一。每个服务内嵌一个轻量的故障注入引擎，通过 REST API 在运行时动态开关，不需要重启服务。

#### ChaosMonkey.java（单例 Bean，线程安全）

```java
@Component
public class ChaosMonkey {

    private final AtomicBoolean enabled = new AtomicBoolean(false);

    // 故障注入配置
    private volatile int latencyMs = 0;           // 注入的额外延迟（毫秒）
    private volatile int latencyJitterMs = 0;     // 延迟抖动范围（毫秒）
    private volatile double exceptionRate = 0.0;  // 异常抛出概率 0.0-1.0
    private volatile String exceptionMessage = "Chaos monkey injected fault";
    private volatile double httpErrorRate = 0.0;  // 返回 HTTP 500 的概率 0.0-1.0
    private volatile int httpErrorCode = 500;     // 返回的 HTTP 错误码

    private final Random random = new ThreadLocalRandom.current();

    // 在业务方法入口处调用此方法
    public void maybeInjectFault() {
        if (!enabled.get()) return;

        // 注入延迟
        if (latencyMs > 0) {
            int jitter = latencyJitterMs > 0 ? random.nextInt(latencyJitterMs) : 0;
            try {
                Thread.sleep(latencyMs + jitter);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }

        // 注入异常
        if (exceptionRate > 0 && random.nextDouble() < exceptionRate) {
            throw new RuntimeException(exceptionMessage);
        }
    }

    // 检查是否应该返回 HTTP 错误（在 Controller 层调用）
    public Optional<Integer> maybeHttpError() {
        if (!enabled.get()) return Optional.empty();
        if (httpErrorRate > 0 && random.nextDouble() < httpErrorRate) {
            return Optional.of(httpErrorCode);
        }
        return Optional.empty();
    }

    // getter/setter/reset 方法...
    public ChaosConfig getConfig() { ... }
    public void updateConfig(ChaosConfig config) { ... }
    public void reset() { /* 恢复所有默认值，disabled */ }
}
```

#### ChaosController.java（故障注入 REST API）

```java
@RestController
@RequestMapping("/chaos")
public class ChaosController {

    private final ChaosMonkey chaosMonkey;

    // GET /chaos/status — 查看当前故障注入状态
    @GetMapping("/status")
    public ChaosConfig getStatus() { ... }

    // POST /chaos/enable — 启用故障注入
    @PostMapping("/enable")
    public ResponseEntity<String> enable() { ... }

    // POST /chaos/disable — 禁用故障注入（恢复正常）
    @PostMapping("/disable")
    public ResponseEntity<String> disable() { ... }

    // POST /chaos/latency — 注入延迟
    // Body: {"latencyMs": 3000, "jitterMs": 1000}
    @PostMapping("/latency")
    public ResponseEntity<String> setLatency(@RequestBody LatencyConfig config) { ... }

    // POST /chaos/exception — 注入异常
    // Body: {"rate": 0.5, "message": "Database connection timeout"}
    @PostMapping("/exception")
    public ResponseEntity<String> setException(@RequestBody ExceptionConfig config) { ... }

    // POST /chaos/http-error — 注入 HTTP 错误码
    // Body: {"rate": 0.3, "statusCode": 503}
    @PostMapping("/http-error")
    public ResponseEntity<String> setHttpError(@RequestBody HttpErrorConfig config) { ... }

    // POST /chaos/reset — 重置所有故障注入（关闭并清除配置）
    @PostMapping("/reset")
    public ResponseEntity<String> reset() { ... }

    // POST /chaos/scenario/{name} — 预置故障场景一键注入
    // 支持的 scenario: "slow-db", "redis-timeout", "cascade-failure", "high-error-rate"
    @PostMapping("/scenario/{name}")
    public ResponseEntity<String> applyScenario(@PathVariable String name) { ... }
}
```

**预置故障场景说明**：

- `slow-db`: 启用 3000ms + 1000ms jitter 的延迟，模拟数据库慢查询
- `redis-timeout`: 启用 5000ms 延迟，模拟 Redis 连接超时
- `cascade-failure`: 启用 80% 异常率，message="Connection refused"，模拟级联故障
- `high-error-rate`: 启用 50% HTTP 503 错误率，模拟服务过载

### 3. Java 微服务详细设计

两个 Spring Boot 3.x + JDK 17 微服务。**不在代码中引入任何 OTel trace/metrics SDK 依赖**，全部通过 javaagent 零代码注入。

#### order-service（端口 8080）

**API 接口**:

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/orders` | 查询所有订单（分页，DB 查询 trace） |
| GET | `/api/orders/{id}` | 查询单个订单（Redis 缓存 + DB fallback） |
| POST | `/api/orders` | 创建订单（跨服务调用 + DB 写入 + 缓存操作） |
| GET | `/api/orders/user/{userId}` | 按用户查询订单（DB 查询 trace） |
| GET | `/api/orders/slow` | 模拟慢查询（固定延迟，不受 chaos 控制） |
| GET | `/chaos/status` | 查看故障注入状态 |
| POST | `/chaos/enable` | 启用故障注入 |
| POST | `/chaos/disable` | 禁用故障注入 |
| POST | `/chaos/latency` | 注入延迟 |
| POST | `/chaos/exception` | 注入异常 |
| POST | `/chaos/http-error` | 注入 HTTP 错误码 |
| POST | `/chaos/reset` | 重置所有故障注入 |
| POST | `/chaos/scenario/{name}` | 一键注入预置故障场景 |

**业务逻辑**:

1. `GET /api/orders/{id}`：先查 Redis（key: `order:{id}`，TTL 60s），miss 则查 MySQL 并回填缓存
2. `POST /api/orders`：
   - 调用 `chaosMonkey.maybeInjectFault()` — 故障注入检查点
   - REST 调用 inventory-service `GET /api/inventory/{sku}` 检查库存和价格
   - REST 调用 inventory-service `POST /api/inventory/reserve` 扣减库存
   - INSERT 订单到 MySQL，total_price = price × quantity
   - 删除相关 Redis 缓存
   - 捕获所有异常，返回合适的 HTTP 错误码
3. `GET /api/orders/slow`：Thread.sleep(2000-5000ms 随机)，模拟慢查询，用于演示延迟 trace

**关键日志点（SLF4J Logger）**:
```java

log.info("Cache MISS for order id={}, querying database", id);

log.info("Calling inventory-service to check stock for sku={}", sku);
log.info("Calling inventory-service to reserve stock for sku={}", sku);
log.info("Order created successfully: orderId={}, totalPrice={}", order.getId(), order.getTotalPrice());
log.warn("Inventory check failed for sku={}: {}", sku, response.getStatusCode());
log.error("Order creation failed for user={}, sku={}: {}", userId, sku, e.getMessage());
```

#### inventory-service（端口 8081）

**API 接口**:

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/inventory` | 查询所有库存 |
| GET | `/api/inventory/{sku}` | 查询单品库存（Redis 缓存 + DB fallback） |
| POST | `/api/inventory/reserve` | 扣减库存（DB 更新 + 缓存更新） |
| GET | `/chaos/status` | 查看故障注入状态 |
| POST | `/chaos/enable` | 启用故障注入 |
| POST | `/chaos/disable` | 禁用故障注入 |
| POST | `/chaos/latency` | 注入延迟 |
| POST | `/chaos/exception` | 注入异常 |
| POST | `/chaos/http-error` | 注入 HTTP 错误码 |
| POST | `/chaos/reset` | 重置所有故障注入 |
| POST | `/chaos/scenario/{name}` | 一键注入预置故障场景 |

**业务逻辑**:

1. `GET /api/inventory/{sku}`：先查 Redis（key: `inventory:{sku}`，TTL 120s），miss 则查 MySQL 并回填缓存
2. `POST /api/inventory/reserve`：
   - 调用 `chaosMonkey.maybeInjectFault()` — 故障注入检查点
   - 查询当前库存
   - 库存不足返回 409 Conflict + JSON body `{"error": "Insufficient stock", "available": n, "requested": m}`
   - 库存充足则更新数量，更新 Redis 缓存
3. 额外：在所有 GET 请求中加入 30-200ms 随机延迟，模拟真实 DB 查询耗时

**关键日志点**:

log.warn("Insufficient stock: sku={}, available={}, requested={}", sku, available, requested);

### 8. Grafana

使用 `grafana/grafana:latest`。

**datasources.yaml — 三大信号关联跳转（最关键的配置）**:

```yaml
apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    jsonData:
      exemplarTraceIdDestinations:
        - name: traceID
          datasourceUid: tempo
      httpMethod: POST

  - name: Tempo
    type: tempo
    access: proxy
    uid: tempo
    url: http://tempo:3200
    jsonData:
      tracesToLogsV2:
        datasourceUid: loki
        spanStartTimeShift: "-1h"
        spanEndTimeShift: "1h"
        filterByTraceID: true
        filterBySpanID: false
        tags:
          - key: "service.name"
            value: "service_name"
      tracesToMetrics:
        datasourceUid: prometheus
        spanStartTimeShift: "-1h"
        spanEndTimeShift: "1h"
        tags:
          - key: "service.name"
            value: "service_name"
        queries:
          - name: "Request rate"
            query: "sum(rate(http_server_request_duration_seconds_count{$$__tags}[5m]))"
          - name: "Error rate"
            query: "sum(rate(http_server_request_duration_seconds_count{$$__tags,http_response_status_code=~\"5..\"}[5m]))"
      nodeGraph:
        enabled: true
      serviceMap:
        datasourceUid: prometheus

  - name: Loki
    type: loki
    access: proxy
    uid: loki
    url: http://loki:3100
    jsonData:
      derivedFields:
        - name: TraceID
          datasourceUid: tempo
          matcherRegex: "traceID=(\\w+)"
          url: "$${__value.raw}"
          matcherType: "regex"
```

### 10. 流量生成器

```yaml
load-generator:
  image: alpine/curl
  depends_on:
    order-service:
      condition: service_healthy
  restart: always
  entrypoint: ["/bin/sh", "-c"]
  command:
    - |
      echo "Waiting 15s for services to stabilize..."
      sleep 15
      echo "Starting continuous load generation..."
      while true; do
        # 查询所有订单
        curl -s http://order-service:8080/api/orders > /dev/null 2>&1

        # 查询单个订单（触发 Redis 缓存命中/未命中）
        ID=$$(shuf -i 1-5 -n 1)
        curl -s http://order-service:8080/api/orders/$$ID > /dev/null 2>&1

        # 按用户查询
        curl -s http://order-service:8080/api/orders/user/user-001 > /dev/null 2>&1

        # 创建订单（跨服务调用 + DB + Redis）
        SKU="SKU-00$$(shuf -i 1-4 -n 1)"
        QTY=$$(shuf -i 1-3 -n 1)
        curl -s -X POST http://order-service:8080/api/orders \
          -H "Content-Type: application/json" \
          -d "{\"userId\":\"user-$$(shuf -i 1-10 -n 1)\",\"sku\":\"$$SKU\",\"quantity\":$$QTY}" \
          > /dev/null 2>&1

        # 尝试购买库存为 0 的商品（触发 409）
        curl -s -X POST http://order-service:8080/api/orders \
          -H "Content-Type: application/json" \
          -d '{"userId":"user-999","sku":"SKU-005","quantity":1}' \
          > /dev/null 2>&1

        # 调用慢接口
        curl -s http://order-service:8080/api/orders/slow > /dev/null 2>&1 &

        # 直接查 inventory 服务
        curl -s http://inventory-service:8081/api/inventory > /dev/null 2>&1
        curl -s http://inventory-service:8081/api/inventory/SKU-001 > /dev/null 2>&1

        sleep 3
      done
```

## 重要注意事项

1. Java 微服务除了 Logback OTel Appender 之外，**不引入任何 OTel SDK 依赖**，trace 和 metrics 全部由 javaagent 自动注入
2. OTel Java Agent 会自动 instrument JDBC（MySQL）、Lettuce（Redis）、RestTemplate（HTTP），在 trace 中你会看到 SQL 语句（已脱敏）和 Redis 命令的独立 span
3. ChaosMonkey 故障注入是纯应用层实现，不依赖任何外部工具，通过 REST API 运行时动态控制
4. 所有配置文件必须完整可用，不能有占位符或省略号
5. 确保 Grafana datasources.yaml 中三大数据源的关联跳转配置完整
6. docker-compose 中 healthcheck 要配准确，避免服务依赖导致启动失败
7. MySQL init.sql 挂载到 `/docker-entrypoint-initdb.d/` 目录自动初始化
